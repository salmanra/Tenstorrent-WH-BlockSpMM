DDA vs cuSPARSE
Sparse Regime
In sparse regimes, cuSPARSE encounters a performance failure mode due to its inability to omit unnecessary compute. The entire output matrix is computed, even the zero blocks, which dominate the runtime at low density. Our algorithm assigns work only to cores which are responsible for nonzero output blocks of the output dense matrix. Still, as the number of nonzero rows in the input increases, cuSPARSE’s SpMM algorithm achieves greater performance, eventually matching the performance of our Wormhole algorithm for most sparsity patterns. The density point at which cuSPARSE matches the performance of Wormhole greatly depends on the input block size and the sparsity pattern of the input matrix. The Row and Random sparsity patterns are favorable to Wormhole’s dynamic data sharing capabilities and the input BSR format, so Wormhole carries its advantage across all densities. The Banded test cases expose a weakness in our DDA algorithm’s ability to take advantage of data sharing opportunities, while the Column test cases are well-suited to cuSPARSE’s column-major internal representation, which allows cuSPARSE’s performance to dominate those test cases as density increases.

TT-Metal’s kernel APIs expose 32x32 block sizes as the smallest unit of matrix multiplication acceleration. GEMM and SpMM kernels which use the matmul_tiles API exhibit significant performance improvements as block size increases as the data movement pipeline more easily saturates the compute units because each matrix engine has much more work to do for each block. This performance improvement with block size holds up to a block size of 256x256, the largest power of two block size which allows for equal sizing of in0 and in1 blocks under the double-buffered scheme. GPU kernels exhibit similar performance scaling with block size, but due to the Tensor Core’s single-cycle matmul size being smaller than that of the matrix engine, the optimal block size on a Tensor core is also smaller, with the cuSPARSE and cuBLAS documentation stating that block sizes of 64x64 are the smallest near-optimal sizes for performance. We see this discrepancy in optimal block size play out in our experiments, with cuSPARSE significantly outperforming our Wormhole algorithms in sparse cases when the block size was 32x32.

Semi-sparse Regime
Semi-sparse test cases, which we define here as containing sparse matrices with densities ranging from 5% to 50%, allow us to assess how closely our algorithms approach the performance of optimized GEMM kernels while still contending with the challenges and advantages of sparse matrix formats. Much like the sparse cases, our Wormhole algorithm performs the best on the Row and Random sparsity patterns. In the semi-sparse case, the performance gap between our Wormhole algorithm on the different sparsity patterns shrinks as density increases. This behavior is to be expected because many qualities of the sparse matrices become very similar as the density increases: the average row size and column size, the number of rows which process the same dense matrix block at a given iteration, and load-imbalance are a few performance-defining qualities which tend to “converge” as density increases. In contrast, cuSPARSE performance remains sensitive to input sparsity pattern even at densities of up to 50%. With block size of 256x256, our Wormhole algorithm outperforms cuSPARSE in the Row and Random test cases, while cuSPARSE holds the advantage in the Banded and Column test cases.
DDA vs SnF
In our testing, the use of store-and-forward for sharing blocks of the sparse matrix across the core grid was nearly unilaterally beneficial to performance. In exchange for incurring a message-depth cost of 8 hops, store-and-forward attains optimally minimal DRAM read count for the sparse matrix and is guaranteed to use the fewest number of hops to share blocks of the sparse matrix across the core grid. This is all made possible because the sparse input matrix is stored in BSR format, ensuring that any 2D, stationary-C-like algorithm will have perfect data sharing of the sparse matrix across rows of the Tensix grid. None of these facts hold true for data movement of the dense matrix, so there are cases where using DDA for data movement of the dense matrix is detrimental to overall performance. Specifically, in cases where DDA exposes little to no data sharing, the reduction in total DRAM reads is too small to offset the runtime share-set discovery and Tensix grid column-wide barrier synchronizations between blockwise matrix multiplies. Since the input matrix is in BSR format, low density test cases exhibit a low count in the number of output rows of the dense matrix. This mitigates the synchronization overhead of DDA and results in great performance in all but the most degenerate cases. For some cases, the sparse matrix is so sparse and exhibits so few data sharing opportunities that naive data movement achieves optimal or near optimal DRAM read count. In such cases, both store-and-forward and dynamic direct addressing suffer performance degradation due to their runtime synchronization overheads. A simple example in which naive data movement outperforms is the pure diagonal case. In this case, each Tensix tile performs a single block matrix multiplication, so the data movement pipeline never has a chance to set up, and SnF and DDA never get to hide their synchronization overheads with compute, despite realizing reductions in DRAM read count on the sparse matrix.

Finally, we turn to triangular matrices to expose a weakness of the DDA data movement scheme. In semi-sparse test cases in which DDA cannot find data sharing opportunities, the share-set discovery and column-wide barrier overheads degrade performance significantly when compared to naive data movement of dense matrix blocks. This is apparent in the case of upper-triangular matrices, where DDA incurs the same number of DRAM reads as naive data movement while exhibiting high synchronization overhead.

These weaknesses in the data movement schemes are artifacts of generality and simplicity: these schemes require no preprocessing as they are and demonstrate the ability of the dual-NoC architecture to hide sparse indexing overhead with computation when the sparse matrix is amenable to pipelined execution. The TT-Metal programming paradigm is equipped to face the challenges posed by diagonal and upper-triangular matrices, but this work focuses on exploiting only the structure of arbitrary BSR matrices.
Effects of Load Balancing
Since our data movement schemes are designed for stationary-C algorithms for SpMM, we partition the output matrix into blocks and assign each output block to a single Tensix tile. Since on-chip SRAM is limited, we have to limit the size of the output blocks. This fact then opens up a design question. When there are more than one output blocks for each Tensix tile, how do assign those blocks across the Tensix grid? To start, first let any two output blocks in the same output row be scheduled on Tensix tiles in the same grid row. The number of block matrix multiplies required to compute an output block is equal to the number of nonzero blocks in the block row of the output block. Since the sparse matrix is in BSR format, it is this design decision that makes store-and-forward such a consistent strategy for improving performance. Our load balancing algorithm is especially impactful when combined with our DDA data movement scheme because it balances workload by row in the manner discussed above and it balances work by column. The latter point can be illustrated by the following argument. Consider our optimized kernel which uses store-and-forward for the sparse matrix and DDA for the dense matrix. The column-wide barriers following each block matrix multiply impose an implicit barrier across the entire Tensix grid because the pipeline of reads of the sparse matrix only unblocks once the compute kernel has completed its matrix multiplication on a set of input blocks, which includes the input block of the dense matrix whose data movement engages in the DDA column-wide synchronization. Therefore, our optimized kernel implicitly processes output blocks in ordered lists, and a Tensix tile cannot move on to the next output block in its list until all other Tensix tiles have completed the computation of their output block at that index in their lists. For the sake of this discussion, we use “iteration” to the set of output blocks processed by the Tensix grid concurrently. Our load-balancing algorithm ensures that the amount of work each Tensix tile performs within a single iteration is optimally balanced.
